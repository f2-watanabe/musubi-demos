グルメサイト予約取込手順

食べログ・ホットペッパー等のグルメサイト経由で入った予約を、その予約確定通知メールから自動で Musubi のネット予約台帳（reservations）に反映する仕組みのセットアップ・運用手順。

1. 仕組み

グルメサイトの予約確定通知メール
  → 店舗ごとの取込アドレス（yoyaku-<slug>@in.musubiweb.com）
  → Cloudflare Email Routing
  → Email Worker（musubi-mail-inbox）が本文を解析
  → admin API（POST /api/reserve/mail-inbox）
  → 日時・人数が読み取れれば reservations に自動登録（channel = tabelog / hotpepper 等）
     読み取れない・キャンセル/変更通知の場合は運営（LINE・メール）に通知し、手動で台帳反映
  • 取込アドレスの <slug> は店舗の site_project（builderのURLに使っているのと同じスラッグ）。
  • 自動登録された予約は、admin/portal の予約一覧で通常予約と区別できるよう「食べログ」「ホットペッパー」等のチャネルバッジが付く。
  • 本文の書式が変わる・見たことのない書式のメールが来る等でパースに失敗しても、そのメールは inbound_mails に記録され運営へ通知が飛ぶ。自動反映に失敗しても取りこぼしにはならず、必ず運営が気づいて手動で台帳に反映できる。
  • キャンセル・予約変更の通知メールは（書式が未確定なため）自動処理せず、常に運営通知に回して手動対応する。
  • 送信元認証（SPF/DKIM）ゲート: Email Worker はメールごとに SPF/DKIM のいずれかが pass したか（auth_pass）を判定して admin API に渡す。auth_pass が false のメール（旧Workerからの未対応リクエストを含む）は、本文のパースに成功していても自動登録されず、手動対応キュー（inbound_mails 記録＋運営メール通知、理由「送信元認証に失敗したメール」）に回る。なりすましメールによる不正な予約登録を防ぐための仕様。
  • 確認方法: admin の inbound_mails テーブル（または管理画面があればそこ）で該当メールの status を見る。unparsed かつ運営への通知メール件名が「グルメサイト予約メール: 手動で台帳反映してください（送信元認証に失敗したメール）」なら SPF/DKIM ゲートで止まったと判別できる。本文が正しく読み取れているのに反映されない場合は、まずここを疑う（食べログ/ホットペッパー側の送信ドメインでSPF/DKIMが正しく通っているかを確認）。

2. 店舗ごとのセットアップ

対象店舗の取込アドレスは次の形式。

yoyaku-<site_project>@in.musubiweb.com

<site_project> は admin.musubiweb.com/stores/<id> で確認できる店舗のスラッグ。

食べログの場合

食べログの店舗会員ページでは、予約通知の送付先メールアドレスを最大3件まで登録できる。

  1. 食べログ店舗会員ページにログインし、予約通知メールの設定画面を開く。
  2. 送付先アドレスの1つに yoyaku-<site_project>@in.musubiweb.com を追加する（既存の店舗宛アドレスは残したままでよい）。
  3. 保存する。

ホットペッパー（レストランボード）の場合

レストランボード側で通知先アドレスを追加できるか現時点で未確認。追加できない場合は、既存の通知メール受信先（店舗のGmail等）から自動転送で取込アドレスへ流す代替手順を使う。

  1. レストランボードの通知メール設定画面で、通知先アドレスを追加登録できるか確認する。
  • 追加できる場合: 食べログと同様に yoyaku-<site_project>@in.musubiweb.com を追加登録する。
  1. 追加できない場合（代替手順・Gmail転送の例）:
  2. 通知メールが届いているGmail（またはその他メールサービス）の設定を開く。
  3. 「メール転送とPOP/IMAP」→「転送先アドレスを追加」で yoyaku-<site_project>@in.musubiweb.com を登録し、確認コードで有効化する。
  4. フィルタを作成し、差出人が jp_kanri@hotpepper.jp のメールを自動転送するルールを追加する（他の用途のメールまで転送されないよう、差出人条件を必ず付ける）。
  5. 転送設定後、実際に届いたホットペッパー通知が転送されることを確認する。

3. 動作確認チェックリスト

[ ] 対象店舗宛にテスト予約を入れる（食べログ/ホットペッパーの実際の予約フローで確定させる）
[ ] 数分以内に admin の予約一覧（該当店舗）に「食べログ」または「ホットペッパー」バッジ付きで予約が反映されることを確認する
[ ] 店主のLINEに「◯◯予約 取込: 日時 人数 名前様」の通知が届くことを確認する（LINE自動予約導入店舗のみ）
[ ] 意図的に読み取りづらい内容（あるいは実際のキャンセル通知）を送り、運営（LINE通知・ALERT_EMAILメール）に「手動で台帳反映してください」の通知が届くことを確認する

4. 注意

  • 本文書式が変わると自動取込に失敗する。 グルメサイト側が通知メールのテンプレートを変更した場合、日付・時刻・人数のいずれかが読み取れなくなり自動登録は止まる。ただしその場合も必ず運営通知（LINE + ALERT_EMAIL）に回るため、取りこぼしは発生しない。通知が増えてきた場合はパーサー（musubi-admin/lib/inboundMail.ts）の表記ゆれ対応を見直す。
  • 自動登録された予約は容量（満席）チェックを行わない。グルメサイト側で既に確定済みの予約を後追いで台帳に記録するだけのため、必ず記録を優先する。
  • 同じ通知メールが複数回届いても（グルメサイト側の再送等）、予約番号または件名・本文の一致で重複登録を防ぐ。
