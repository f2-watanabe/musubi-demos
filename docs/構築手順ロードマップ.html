環境構築ロードマップ（細粒度）

プラットフォーム（管理アプリ・顧客サイト量産・口コミ自動返信・月次レポート・課金）を、振り返り可能な手順で構築するための実装ロードマップ。

サイト制作・運用ロードマップ は「1店舗の受注〜公開〜運用」の粒度。本書はその土台となる環境そのものの構築を、依存関係・受け入れ基準・残す証跡まで落とした粒度で扱う。

  • 進め方: 上から順に。ただし Phase 0 のGBP API申請は最優先で即着手（審査リードタイムが全体の律速）。
  • 各ステップは - [ ] のチェックで進捗管理。完了時は 08_環境構築/構築ログ.md に証跡（日付・コミット・結果）を残す。
  • 「受け入れ基準」を満たすまで次フェーズに進まない。

全体像（依存関係）

Phase 0 アカウント/API申請 ──▶ Phase 1 口コミ自動返信疎通 ──▶ Phase 6 承認UI統合
        │                                   ▲
        ├──▶ Phase 2 データ基盤/認証 ───────┘
        │            │
        │            ├──▶ Phase 3 顧客サイト量産基盤
        │            ├──▶ Phase 4 月次レポート自動集計
        │            └──▶ Phase 5 課金(Stripe)
        └──▶ Phase 7 監視・バックアップ（全体に横断）

Phase 0 ・ アカウント準備とAPI申請（最優先）

目的: 後工程の前提となる外部アカウントと、リードタイムの長いAPI審査を先に片付ける。

手順

  • ✓独自ドメイン musubiweb.com 取得
  • ✓Google Workspace 開設・ドメイン認証 → admin@musubiweb.com 発行（以降の各SaaSはこのアカウントで登録）
  • Google Cloud プロジェクト作成（請求先アカウント紐付け）
  • Business Profile API（旧 My Business API）の利用申請を提出 ← 審査に日数がかかるため最初に出す
  • OAuth 同意画面を構成（スコープ business.manage、テスト→本番公開）
  • OAuth クライアントID/シークレット発行（リダイレクトURI登録）
  • Anthropic アカウント・APIキー発行（請求上限を設定）
  • Supabase プロジェクト作成（リージョン: Tokyo）
  • Vercel アカウント・チーム作成（GitHub連携）
  • Cloudflare アカウント作成（Pages・Registrar）
  • Stripe アカウント作成（本人確認・入金口座登録）
  • Resend アカウント作成（送信ドメイン認証）
  • GA4 プロパティ作成・データストリーム発行

使用サービス: Google Cloud / Anthropic / Supabase / Vercel / Cloudflare / Stripe / Resend / GA4

受け入れ基準: GBP API申請の受付完了。各SaaSにログインでき、キー類を安全に保管（パスワードマネージャ／Vercel・Supabaseのシークレット）。

残す証跡: 申請日・申請ID、各サービスのプロジェクト名／ID、発行キーの保管場所（値そのものは記録しない）。

Phase 1 ・ 口コミ自動返信の疎通（中核・最優先で検証）

目的: 月額価値の根幹「口コミ自動返信」が技術的に成立することを、実店舗1件で実証する。

前提: Phase 0 のGBP API承認・OAuth設定。検証用に協力してくれる実店舗1件（自店・知人店可）。

手順

  • ✓プロトタイプ gbp-review-reply 実装（取得→下書き→承認→投稿）
  • ✓mockモードでパイプライン疎通を確認
  • ✓セキュリティ対策（インジェクション／CSRF／無人公開投稿の遮断）
  • 対象店舗オーナーからマネージャー権限の招待を受ける
  • npm run auth でOAuth認可 → token.json 取得
  • 対象ロケーションIDを取得し .env に設定
  • GBP_MODE=live で口コミ取得を確認
  • Claude下書きの品質を実口コミで確認（トーン設定の調整）
  • 承認 → 実際にGoogleへ返信が投稿されることを確認（1件）
  • 低評価（★1〜3）が自動投稿されず人間レビューに回ることを確認

使用サービス: Google Business Profile API / Anthropic / （ローカル実行）

受け入れ基準: 実店舗で「取得→下書き→承認→投稿」が1サイクル通る。低評価は自動投稿されない。

残す証跡: 投稿された返信のスクショ、調整したトーン設定、out/run-*.json。

Phase 2 ・ データ基盤と認証

目的: 顧客・契約・口コミ・請求を一元管理するDBと、運営/顧客の認証を用意する。

手順

  • Supabase スキーマ設計（stores / contracts / reviews / invoices / reports）
  • マイグレーション管理を導入（SQL or supabase migrations）
  • RLS（行レベルセキュリティ）方針を決める（顧客は自店のみ閲覧）
  • Auth 設定（運営=管理アプリ、顧客=ポータル。ロール分離）
  • 管理アプリ（Next.js）を Vercel に雛形デプロイ（ダッシュボードの枠だけ）
  • Supabase 接続・環境変数を Vercel に設定

使用サービス: Supabase（Postgres/Auth）/ Vercel / Next.js

受け入れ基準: 管理アプリからログインでき、DBに1件の店舗レコードをCRUDできる。顧客ロールは他店データが見えない。

残す証跡: スキーマ図／DDL、RLSポリシー、デプロイURL。

Phase 3 ・ 顧客サイト量産基盤

目的: content.json + テンプレHTML で店舗サイトを量産し、Cloudflare Pages へ自動デプロイできる状態にする。

手順

  • 既存 restaurant テンプレを基盤として整理
  • 2業種目のテンプレを追加（美容室 or 整体）し量産性を確認
  • GA4・イベント計測（フォーム送信／電話タップ）をテンプレに標準同梱
  • build_site.py を管理アプリのビルド機能として接続（or CI化）
  • Cloudflare Pages へのデプロイ自動化（ドメイン割当・SSL）
  • 独自ドメイン取得〜割当の手順を確立（Registrar）

使用サービス: Cloudflare Pages / Registrar / GitHub Actions（任意）/ GA4

受け入れ基準: 新規店舗のJSONを用意 → 1コマンド／1操作で公開サイトがHTTPSで立ち上がり、計測タグが動く。

残す証跡: 追加テンプレのプレビュー、デプロイ手順、計測の動作確認スクショ。

Phase 4 ・ 月次レポート自動集計

目的: 解約防止の武器「月次レポート」を、手作業を最小化して毎月生成・公開する。

手順

  • GA4 Data API でアクセス/イベント数を取得する処理
  • GBP API で口コミ件数・平均評価・返信対応状況を集計する処理
  • 既存 04_サイトテンプレート/report/ テンプレへデータ流し込み
  • 顧客ポータルへ公開する導線（更新通知メールは Resend）
  • 月次の定期実行（Vercel Cron）
  • 人の手は「ひとことコメント」のみに限定

使用サービス: GA4 Data API / GBP API / Vercel Cron / Resend

受け入れ基準: 1店舗ぶんのレポートが自動でデータ取得〜生成〜ポータル公開まで通る。手作業はコメント追記だけ。

残す証跡: 自動生成されたレポート、Cron実行ログ。

Phase 5 ・ 課金（Stripe サブスク）

目的: 月額の自動回収と未入金対応を仕組み化し、請求業務をゼロに近づける。

手順

  • Stripe 商品・料金（ライト/スタンダード/プレミアム）を登録
  • サブスクリプション作成フロー（契約時に課金開始）
  • 初期費の決済リンク
  • Webhook 受信（支払い成功/失敗/解約）→ Supabase へ反映
  • カード期限切れ・失敗時のリトライ＆案内（Resend）
  • 未入金フロー（リマインド→一時停止予告→停止）の自動化
  • 領収書／インボイスの自動発行

使用サービス: Stripe / Vercel（Webhook）/ Supabase / Resend

受け入れ基準: テストモードで「契約→毎月課金→失敗時リトライ→解約」が一通り動き、状態がDBに同期される。

残す証跡: Stripeダッシュボードのテスト履歴、Webhookログ、状態遷移の確認。

Phase 6 ・ 顧客ポータルへ承認UI統合

目的: 口コミ承認をCLIからWeb（顧客ポータル）へ移し、店舗オーナー自身が承認できるようにする。

手順

  • ポータルに「口コミ承認」画面を実装（一覧・下書き表示・修正・承認/差戻し）
  • 承認アクション → GBP API へ投稿（サーバ側で実行）
  • 低評価・検証フラグ付きは明示し、慎重な承認を促すUI
  • 新着口コミの定期取得（Cron）→ 未対応をポータルに表示
  • 対応状況を月次レポートへ連携

使用サービス: Next.js（ポータル）/ Supabase / GBP API / Vercel Cron

受け入れ基準: 店舗オーナーがポータル上で下書きを確認→承認→投稿でき、低評価には警告が出る。

残す証跡: 画面スクショ、承認〜投稿の通しテスト。

Phase 7 ・ 監視・運用・バックアップ（横断）

目的: 公開後に落ちない・データを失わない運用基盤。

手順

  • UptimeRobot で公開サイト・主要エンドポイントの死活監視
  • エラー通知（メール/Slack等）
  • Supabase の定期バックアップ確認
  • APIキー・トークンの保管と棚卸し（漏洩時の失効手順）
  • 簡易ランブック（障害時の一次対応）を 構築ログ 隣に用意

使用サービス: UptimeRobot / Supabase / Vercel

受け入れ基準: 監視が通知を出せる。バックアップから復元できることを1度確認。

残す証跡: 監視設定、復元テストの記録。

完了の定義（環境構築フェーズのゴール）

  • 実店舗1件で「サイト公開 → 口コミ自動返信 → 月次レポート公開 → 月額課金」が人手最小で一周する。
  • 同じ手順で2店舗目を、前回より短時間で立ち上げられる（量産性の確認）。

ここまで到達したら、営業で獲得した顧客を本番投入してスケールフェーズへ移る。
