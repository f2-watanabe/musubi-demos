自社タスク一覧 — 手動対応が必要な事項

運用手順書_申込からクロージングまで.md の中で、コード（管理アプリ／ポータル）だけでは完結せず、外部サービスの設定や手作業が必要な事項をまとめています。運用開始前に上段（初回設定）を、日々の運用で下段（都度対応）を確認してください。

値そのもの（APIキー・トークン・パスワード）はこのファイルに書きません。保管場所（Vercel 環境変数・各サービスのダッシュボード等）だけを記します。

A. 初回設定（運用開始前に一度）

A-1. Stripe（サブスク課金）

  • 本番キーの用意とモード確認: 納品時のサブスク開始に本番の STRIPE_SECRET_KEY を使う。現在、料金プラン（price）は テストモードで作成済み（Vercel の STRIPE_PRICE_LIGHT/STANDARD/PREMIUM）。本番運用時は次のどちらかに揃える:
  • テストで通す間: 本番 STRIPE_SECRET_KEY もテストモードのキーにする。
  • 本番課金する時: musubi-admin/scripts/stripe_setup.mjs を ライブキーで実行してライブの price を作り、Vercel の STRIPE_PRICE_* を差し替える。
  • ※キーとpriceのモードがずれると、納品時のサブスク作成が失敗します。
  • Webhook の設定: STRIPE_WEBHOOK_SECRET は登録済み・稼働中です（2026-07-15）。Stripe ダッシュボードの Webhook エンドポイントは https://admin.musubiweb.com/api/stripe/webhook に設定してあり、署名シークレットを Vercel 環境変数 STRIPE_WEBHOOK_SECRET に登録済みです。入金・失敗ステータスの同期、および初期費用のカード決済（Payment Link）完了時の自動 paid 化が動作します。
  • 顧客のカード登録（Billing Portal）: 顧客ポータルの「お支払い」タブ（/pay）は Stripe の Customer Portal（Billing Portal）を使ってカード登録・変更を行う。これを動かすには、musubi-portal 側の Vercel 環境変数にも STRIPE_SECRET_KEY（admin と同一 Stripe アカウントのキー）を設定し、かつ Stripe ダッシュボードで Customer Portal を有効化しておく必要がある。どちらか未構成だと /pay でのセッション作成が失敗する。※本対応は実施済み。
  • 保管場所: Stripe ダッシュボード、Vercel（musubi-admin・musubi-portal）環境変数。
  • 【本番切替 完了（2026-07-19）】 以下5手順をすべて実施済み。ライブprice 3件（¥5,000/¥15,000/¥20,000・月額）検証済み、ライブWebhook（6イベント・enabled）稼働、署名なしリクエスト400を実測、admin/portal redeploy済み。Preview 環境も本番キーのため、プレビュー上での決済テストは実カード課金になる点に注意（テストしたい場合は Preview のみテストキーに戻す）。
  1. Stripe ダッシュボード（本番モード）で Secret key を取得。
  2. cd musubi-admin && STRIPE_SECRET_KEY=sk_live_... node scripts/stripe_setup.mjs を実行し、本番の price ID 3つを得る。
  3. Vercel（musubi-admin）の production 環境変数 STRIPE_SECRET_KEY・STRIPE_PRICE_LIGHT/STANDARD/PREMIUM を差し替え（musubi-portal 側の STRIPE_SECRET_KEY も同時に）。
  4. Stripe 本番モードで webhook https://admin.musubiweb.com/api/stripe/webhook を作成し、署名シークレットを STRIPE_WEBHOOK_SECRET に設定。Customer Portal（Billing Portal）の本番モード有効化も確認。
  5. admin/portal を redeploy して疎通確認（テスト決済1件→ webhook 反映まで）。

A-2. メール（Resend）

  • 設定済み（対応不要）。送信ドメイン send.musubiweb.com は検証済み、Supabase 認証メール（招待）も Resend 経由。招待メールが迷惑メール／プロモーションタブに入る場合があるため、顧客には「Musubi からのメールを確認してください」と案内すると親切。

A-3. Googleビジネスプロフィール（GBP）

  • API 利用承認: 初回申請（ケース 1-4601000041294）は 2026-07-14 に却下。理由は「申請者メールが、オーナー確認から60日以上経過したプロフィールの管理者であること」の要件未達（プロフィール自体は確認済み・管理者紐付け済み、経過日数のみ不足）。
  • 再申請タスク（2026-09-12 以降）: オーナー確認完了は 2026-07-03〜07-13 の間（Gmail に完了メールなし・構築ログと却下日から特定）。60日ルールの解禁は 9/1〜9/11 のどこかのため、確実な 9/12 以降に admin@musubiweb.com で再申請する。早期化したい場合は、確認から60日超の既存リスティングに管理者追加してもらう。
  • 承認までの運用: 口コミ返信は「AI下書き生成 → ポータルで承認」まで動作し、実投稿は手動（下書きをコピーして GBP に貼り付け）。
  • 承認後: リフレッシュトークン等を Vercel 環境変数に設定し、自動投稿を有効化する。
  • 保管場所: Google Cloud / GBP のケース、Vercel 環境変数。

A-4. ドメイン基盤（Cloudflare）

  • サブドメイン運用（<店舗>.musubiweb.com）は管理アプリの公開・削除操作で自動化済み。
  • 公開時の Cloudflare Pages プロジェクト名は、内部で musubi-<サブドメイン> に自動で一意化されるため、顧客のサブドメイン名（例: test）は自由に付けてよい。以前は汎用的な名前だと *.pages.dev の衝突で公開に失敗することがあったが、この自動一意化により解消済み。
  • 保管場所: Cloudflare ダッシュボード、Vercel 環境変数（CLOUDFLARE_API_TOKEN／ACCOUNT_ID／ZONE_ID／SITE_BASE_DOMAIN）。

A-5. GA4（アクセス解析）【対応済み】

  • コード側は実装済み（T23、2026-07-15）: 公開サイトへの共有測定ID自動注入・eventテンプレのgtagは本番デプロイ時に自動で動く状態。
  • 対応済み: GA4 共有プロパティの測定ID（GA4_SHARED_MEASUREMENT_ID）を Vercel 環境変数に登録済み（2026-07-15）。これ以降に公開・再公開したサイトから計測が開始されます。登録前に公開されたままのサイトは、次回の再公開時点から計測対象になります。
  • 保管場所: Google Analytics（GA4）管理画面、Vercel（musubi-admin）環境変数。

B. 案件ごとに都度対応

B-1. 打ち合わせの日程調整（手順書ステップ3）

  • 打ち合わせ予約は実装済み（C節参照）。管理アプリ /meetings で空き枠を作成しておくと、ログイン済み顧客はポータル /meeting から予約できる。未ログインのリードとはメール・電話で調整する。

B-2. 独自ドメインの紐付け（手順書ステップ7）

  • 顧客が独自ドメインを使う場合、Cloudflare での紐付けは手作業。独自ドメイン運用・移管手順.md に従う。
  • サブドメイン運用（<店舗>.musubiweb.com）のみなら自動なので対応不要。

B-3. 口コミの実投稿（手順書ステップ10）

  • GBP API 承認が下りるまでは、承認済みの返信下書きを手動で GBP に投稿する。

B-4. 解約時の独自ドメイン移管（手順書ステップ11）

  • 「独自ドメイン移管」を選んだ場合の実作業は手動。管理アプリの解約処理画面に表示されるチェックリスト、および 独自ドメイン運用・移管手順.md（パターンA: ドメインごと顧客へ移管）に従う:
  • レジストラでの移管申請
  • ネームサーバの確認
  • Cloudflare ゾーンの移管または顧客アカウントへの移行
  • 移管手数料の請求
  • 「サブドメイン削除」を選んだ場合は、削除は管理アプリが自動で行う（手作業なし）。

C. 実装済み（2026-07-05 追加）

以下は当初「未実装」だったが実装済み:

  • 打ち合わせ予約カレンダー: 管理アプリ /meetings で空き枠を作成、顧客はポータル /meeting から空き枠を選んで予約。二重予約は排他制御で防止、予約RPCはアプリログイン＋DB権限の二重防御。確定時に双方へメール。
  • 顧客サイトのネット予約（内部データベース）: 飲食テンプレート（restaurant/cafe/sushi）は予約フォームが内部予約API（/api/reserve）に直結済み。空き枠は管理アプリのサイト制作（builder）の予約設定で編集、予約一覧・キャンセルは /reservations。
  • 未入金時の自動リマインド（dunning）: 決済失敗で当日リマインドメール＋3日の猶予、猶予超過で日次バッチがサイトを一時停止、入金確認で自動復帰。
  • ポータルのプラン別機能ロック: ライトプランは口コミAI返信・月次レポートを非提供（「スタンダードプラン以上でご利用いただけます」表示）。画面・サーバー両方でロック。
  • 申込フォームのレート制限: 同一IPから短時間の連投を制限（隠しフィールド＋IPレート制限の二段）。
