Musubi環境構築ドキュメント

環境変数一覧と登録手順（musubi-admin / musubi-portal）

作成: 2026-07-12（.env.local 消失事故の復旧を機に整備）

原則

  • 本番の正は Vercel の環境変数（Project → Settings → Environment Variables）。本番アプリは Vercel が実行時に注入するため、ローカルのファイルが壊れても本番は影響を受けない。
  • Vercel で「Sensitive」にした変数は vercel env pull で取り戻せない（空文字で出力される）。ローカル開発用の値は Vercel からの pull に頼らず、下記の「値の取得元」から直接取得して .env.local に記載する。
  • .env.local は両リポとも .gitignore 済み（コミットされない）。値をチャット・ドキュメント・リポジトリに書かない。 このファイルにも変数名と取得場所のみ記載する。
  • 控えの推奨: パスワードマネージャ（1Password 等）に「Musubi env」として保管し、.env.local 再構築時はそこからコピーする。

テスト環境（musubi-test）

  • 目的: E2E / dev 検証を本番 Supabase から隔離する。本番 project (hsyhiiqrrtrbxexqyixi) には書込を一切行わない。
  • project ref: vqflyyoexsxqcfajwoxv（region ap-northeast-1、Musubi's Org 配下）
  • 接続先: https://vqflyyoexsxqcfajwoxv.supabase.co
  • スキーマ: 本番の migrations（0001〜0029）を適用した上で、本番との情報スキーマ差分（information_schema.tables/columns, pg_constraint, pg_indexes, pg_policies, pg_proc, pg_trigger, pg_type(enum), storage.buckets, role grants）をゼロまで補正済み（auth/storage 内部テーブルと統計情報は比較対象外）。0002 baseline snapshot は列定義のみで PK/FK/UNIQUE/CHECK を含まないため、本番の pg_constraint から再構成して補完している。加えて migrations に未追跡だった以下も本番の実体に合わせて再現した: reports.avg_rating/reply_rate・leads.google_rating の numeric 精度、perf 系インデックス7本（contracts_store_idx 等）、set_updated_at トリガー関数と5テーブルの updated_at トリガー、cost_alerts の RLS + cost_alerts_staff_read ポリシー。
  • 仕組み: musubi-admin / musubi-portal の両リポに .env.development.local（git 管理外、.gitignore の .env* で除外済み）を配置。Next.js は next dev 実行時に .env.local より優先してこれを読み込むため、dev 起動時のみ musubi-test に接続し、本番 .env.local は一切変更しない。next build/本番実行時は .env.development.local を読まないため本番挙動に影響しない。
  • seed データ: staff アカウント staff-test@musubiweb.com（staff_members 登録済み）、テスト店舗一式（leads name='test'、stores name='test' template='restaurant' + owner アカウント owner-test@musubiweb.com、projects 1件）。パスワードはランダム生成し .env.development.local の先頭コメントに記録（1Password 等への控えは未実施、必要なら手動で控える）。
  • E2E の規律: E2E・dev ログイン検証は必ず musubi-test を使う。本番データに触れる作業（実店舗の実データ確認等）のみ本番 project を読取で参照する。
  • 再構築が必要になったら: 組織ID取得 → POST /v1/projects（region ap-northeast-1）→ ACTIVE_HEALTHY をポーリング → enum型作成 → 0002_baseline_snapshot.sql → PK/FK/UNIQUE/CHECK 補完 → 0001,0003〜0029 を順に適用 → 本節記載の未追跡差分（numeric精度・perfインデックス・updated_atトリガー・cost_alerts RLS）を適用 → storage.buckets に site-assets（public, 10MB, image/png・jpeg・webp・gif）を追加 → スキーマ差分ゼロを機械確認 → auth admin API で staff/owner 作成 → seed 投入。

登録手順

Vercel へ登録/更新:

cd musubi-admin（または musubi-portal）
npx vercel env add <変数名> production   # 対話で値を入力。preview も必要なら同様に

またはダッシュボード（Project → Settings → Environment Variables）。秘密度の高いもの（上表 Sensitive 推奨）は「Sensitive」を有効化 — ただし有効化すると pull で取り戻せなくなるので、必ず取得元 or パスワードマネージャに控えを残すこと。

ローカル .env.local の再構築:

  1. このファイルの表の「L=◎」を最低限埋める（dev 起動に必須: Supabase 4種 + PORTAL_URL + SITE_BASE_DOMAIN）
  2. 機能開発に応じて該当行を追加
  3. 完成したらパスワードマネージャに控えを更新

現状確認コマンド（値を表示しない）:

awk -F= '/^[A-Z]/ {print $1, (length($2)>2 ? "[値あり]" : "[空]")}' .env.local
npx vercel env ls production

2026-07-12 消失事故の復旧チェックリスト（admin/.env.local）

vercel env pull --environment=development の上書きで、Sensitive 区分の値がローカルから消失した。本番は無傷。以下をローカルに入れ直す（取得元は上表）:

  • SUPABASE_URL（固定値なのですぐ書ける）
  • SUPABASE_SERVICE_ROLE_KEY（Supabase Dashboard）← E2E 再開の必須条件
  • SUPABASE_ACCESS_TOKEN（Supabase Account → Access Tokens で再発行）← migration 適用に必要
  • RESEND_API_KEY / ALERT_EMAIL / RESEND_FROM（メールを dev で触る時）
  • ANTHROPIC_API_KEY / ANTHROPIC_ADMIN_KEY（AI・コスト機能を dev で触る時）
  • STRIPE_WEBHOOK_SECRET / STRIPE_PRICE_×3（課金 dev 時）
  • GA4_SA_JSON / GA4_PROPERTY_ID / GOOGLE_MAPS_SERVER_KEY / CRON_SECRET（該当機能の dev 時）

portal/.env.local は SUPABASE_URL / SUPABASE_SERVICE_ROLE_KEY / STRIPE_SECRET_KEY が同様に不足（メール系は残存）。

再発防止

  • vercel env pull を実行する時は必ず出力先を別ファイルにする（例: vercel env pull .env.vercel-check）。.env.local への直接 pull は禁止。
  • .env.local を変更する前にバックアップ: cp .env.local .env.local.bak（.gitignore に .env* が含まれることを確認済みの上で）。
