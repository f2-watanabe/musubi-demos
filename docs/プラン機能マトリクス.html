プラン機能マトリクス（内部用）

「プランごとに何ができるか」が分かりにくいという声を受けて、コード（musubi-admin / musubi-portal）を正の情報源として機能を粒度細かく棚卸しした内部資料。営業・サポート対応時の一次参照に使う。

読み方の注意: ○＝コード上ゲートされて確実に使える／使えない。「運用のみ」＝コード上は全プラン共通で技術的に制限されていないが、営業・契約・料金上そのプランの標準として案内している機能（スタッフが手動設定すれば他プランでも技術的には動く）。この違いを区別して書いている。

1. 前提：ライン判定とプラン体系

1-1. 飲食／企業の判定（ライン判定）

musubi-portal/lib/businessLine.ts（musubi-admin/lib/contractTemplate.ts に複製あり・要同期）:

RESTAURANT_TEMPLATES = ['restaurant', 'sushi', 'cafe', 'ramen']
テンプレートがこの4つ → restaurant（飲食向け）
それ以外すべて         → general（企業・その他向け）

サイトのテンプレート種別（stores.template）で自動判定される。プラン名（light/standard/…と biz_light/biz_standard/…）そのものではなく、テンプレート種別がラインを決める。飲食向けプラン契約でも general テンプレートを選べば企業向け扱いになりうる（通常運用ではしないが、コード上の制約ではない）。

2. 飲食向けマトリクス

対象: businessLine = restaurant（テンプレート restaurant / sushi / cafe / ramen）。

判定の中心は musubi-portal/lib/planAccess.ts の PLAN_FEATURES:

light:    { reviews: false, reports: false, reserve: false }
standard: { reviews: true,  reports: true,  reserve: true  }
premium:  { reviews: true,  reports: true,  reserve: true  }

standard と premium はこの3フラグでは無差別（コード上は同じ）。両者の差は下表の「更新対応回数」「LINE予約の方式」「独自ドメイン」「LINE導入代行」など、PLAN_FEATURES の外側にある個別ロジック・運用ルールで生まれている。

3. 企業向けマトリクス

対象: businessLine = general（テンプレート restaurant/sushi/cafe/ramen 以外すべて）。

PLAN_FEATURES は biz_light / biz_standard / biz_premium いずれも { reviews: false, reports: false, reserve: false }。企業向けプランは口コミAI返信・月次レポート・ネット予約のいずれも、プランを問わず対象外（musubi-portal 側で businessLine(template) !== 'restaurant' の時点でまず弾かれ、その先の planAllows チェックにすら到達しない二重ガード）。

4. カスタムプランの扱い

  • plans.ts: custom は monthly_fee: 0 initial_fee: 0 が初期値。実額は契約書提示時にスタッフが正整数で個別入力（requirePositiveIntFee()）。Stripe Price IDも都度手入力（resolvePriceId() が price_ 始まりを要求）。
  • 機能面はテンプレートを使わないフルオーダー制作が前提（飲食向けLP「カスタム」／企業向けLP「フルオーダー」）。個別相談で決定するため、本マトリクスの○×は適用されない。
  • 更新対応回数のみ既定値あり: updatesPerMonthByPlan.custom = 1（契約時に明示指定しなければこの値にフォールバック）。

5. 全プラン共通

飲食・企業・カスタムを問わず、以下はプラン非依存でコード上ゲートされていない。

  • セルフ編集（顧客ポータル）— 利用回数無制限、公開履歴からの復元込み
  • 縛りなし（契約は1年自動更新・解約は希望月の30日前までに申し出、というのが CONTRACT_DEFAULTS の標準値。中途解約金の定めはひな形上ない）
  • 死活監視（site_domain を持つ公開中の全店舗が対象）
  • 顧客ポータル本体（メッセージ・請求確認・LINE連携設定）
  • 通知（メール／LINE、店舗ごとに設定した notify_email とLINE連携先へ）
  • 契約書（buildContractDocument() で自動生成。プランに応じて料金・更新回数等が差し込まれる）
  • 請求（Stripeカード決済 or 銀行振込。月額は自動継続請求）

Musubi側作業の範囲: 文言・写真の差し替え等の軽微な更新（updatesClause() の文言）。プラン範囲を超える更新・構造変更は別途見積り（business.html FAQ・ネット予約システムマニュアル.md と同旨）。回数超過分の扱い（追加課金/翌月繰越等）はコード上定義がなく、都度の営業判断。

コードとLPの食い違い一覧

  1. 企業向けプランの名称: 解消済み（2026-07-20）。LPを「シンプル／コーポレート／プレミアム」に統一（コードの正式ラベルは識別用に「企業」接頭辞付きのまま＝意図的）。
  2. LINE自動予約のプラン制約: LP・マニュアル類は「プレミアムプラン限定」と案内しているが、コード（content.reserve.line_auto フラグと store_line_channels 登録）にはプラン制約が存在しない。スタッフが手動設定すればライト/スタンダードでも技術的には動作する。運用ルールとしての制限であり、コードのバグではない。
  3. 公式LINE導入代行・独自ドメインのプラン紐付け: どちらも「プレミアム標準」という案内はLP・手順書・コード上のUIラベルで一致しているが、実装上（siteDomain.ts の検証ロジック、LINE導入代行の手順書）はプランによる強制がなく、運用判断でスタッフが実施する形になっている。機能自体はコードで動くため矛盾ではないが、契約プランと実装の結びつきが「運用ルール」止まりである点は認識しておくべき。
