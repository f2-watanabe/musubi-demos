汎用ライン運用手順書（企業・個人・イベント/プロモ案件）

飲食3プラン（ライト/スタンダード/プレミアム）とは別枠の汎用ライン＝カスタムプランの受注〜クロージングまでの手順書です。対象は飲食以外のコーポレートサイト・個人/クリエイターサイト・イベント/プロモーション（ポップアップストア等の期間限定サイト含む）。

商品定義（呼称・見積レンジ・飲食ラインとの違い）の正は 01_提案・設計資料/利益試算の前提と根拠.md §7 です。金額の考え方はそこを必ず参照し、本書では数値を書き写すのみで独自に変更しません。基盤システム（admin / portal / Stripe / Cloudflare Pages）は飲食ラインと共通で、運用手順書_申込からクロージングまで.md の該当ステップを流用できる箇所は「飲食と共通」と明記します。個別見積・個別 Stripe Price 作成・手組み制作など、汎用ライン特有の分岐だけをこの手順書で詳述します。

  • 管理アプリ（admin）: https://musubi-admin-two.vercel.app （スタッフのみ・ログイン必要）
  • 顧客ポータル（portal）: https://musubi-portal.vercel.app （顧客のみ）
  • Stripe ダッシュボード: https://dashboard.stripe.com

フロー全体像

申込 → ヒアリング → 見積・契約 → Stripe Price 作成 → 制作 → プレビュー承認
  → 公開 → 課金開始 → 月次運用 → 会期終了/解約処理

飲食ラインと同じく、案件の進行は admin の案件詳細画面（/stores/[店舗ID]）のステージレールで管理します。汎用ラインでは以下の3点が飲食と異なります。

  1. 契約前に「見積」という工程が独立して発生する（飲食は定価3プランのため見積工程がない）。
  2. 課金開始前に、契約ごとの Stripe Price を個別に作成する工程が挟まる（飲食は共通の3プランPriceを使い回す）。
  3. 制作がテンプレ+AI生成ではなく custom モード／手組みになる。

1. 申込受付

誰が: 見込み客（企業担当者・個人事業主・イベント主催者）

画面: マーケサイトの企業・イベント向けページ、または紹介経由

  1. マーケサイトの企業・イベント向けページ（LP https://musubiweb.com の企業・イベント向け導線）の相談フォームから問い合わせが入るか、既存顧客・取引先からの紹介で話が来る。
  2. フォーム経由の場合は飲食の相談リードと同じ経路で admin /crm にリードとして登録される（種別は「相談」扱い。汎用ラインは定価プランを持たないため「申込」種別は使わない）。
  3. 紹介経由など、フォームを通らずに話が来た場合は、スタッフが admin /crm から手動でリード（または直接 store）を作成する。会社名・団体名、担当者名、連絡先、案件概要（イベント名・会期・予算感が分かれば記載）を登録する。

確認ポイント:

  • admin /crm にリードが表示される（フォーム経由・手動登録のいずれでも）。
  • 案件概要欄に「汎用ライン案件」であることが分かるメモを残す（担当が交代しても飲食案件と混同しないため）。

2. ヒアリング

誰が: スタッフ（顧客への聞き取りは対面・オンライン打ち合わせ・メールのいずれでも可）

画面: 打ち合わせ（対面/オンライン）＋ admin 案件詳細のメモ欄

飲食用のヒアリングシート（portal /hearing）は予約・メニュー等の飲食店固有項目が中心のため汎用ラインでは使いません。代わりに、以下の項目を打ち合わせ（対面・オンライン・メールのいずれか）で手動ヒアリングし、admin 案件詳細のメモ欄または別途ヒアリング記録ファイルに残します。

  1. 目的・ターゲット — サイトを何のために作るか（採用広報・商品プロモ・イベント告知・ポートフォリオ等）、見せたい相手は誰か。
  2. 会期・公開希望日 — イベント/プロモ案件は会期（開始日必須・終了日 or 順次開催）を最優先で確認する。会期が決まっている案件は納期がすべてに優先する。
  3. 掲載内容 — ページ構成の希望（会社概要・実績・アクセス・商品一覧・グッズ情報等、案件の性質に応じて）。
  4. 支給素材の一覧と権利状態 — ロゴ・写真・原稿・キャラクター素材等、何を先方が支給するか、権利処理が済んでいるかを1点ずつ確認する。版権キャラクター等を扱う案件では「権利処理済み素材の支給」を必ず契約条件に明記するため、この段階で漏れなく洗い出す。当方でキャラクター素材やロゴの生成・流用は行わない。
  5. SNS連携 — 掲載する公式SNSアカウント、サイトからのリンク要否。
  6. 参考サイト — デザイン・構成のイメージに近い既存サイトを2〜3例挙げてもらう。
  7. 会期後のサイト扱い — イベント/プロモ案件は会期終了後にサイトをどうするか（閉鎖／アーカイブとして残す／通常サイトとして継続）を契約前にこの時点で方向性を握っておく（最終決定は3章の契約時)。
  8. 月額保守に含める作業 — 保守に何を含めるか（軽微な文言修正・画像差し替えの回数、レポート要否等）を確認し、月額の見積根拠にする。

確認ポイント:

  • 上記8項目がすべて聞き取れている（特に会期・支給素材の権利状態は見積・契約に直結するため未確認のまま次工程に進まない）。
  • ヒアリング内容を admin 案件詳細のメモまたはヒアリング記録ファイルに文書化し、見積担当が読めば追加確認なしで見積を組めるレベルの粒度にする。

3. 見積・契約

誰が: スタッフ

画面: 社内（見積書作成）→ admin /stores/[店舗ID]「契約条件」フォーム → portal /contract

  1. 利益試算の前提と根拠.md §7 の算式で見積を組み立てる。
  • 初期制作費 = 想定工数 × 時間単価 + 外部実費（ドメイン・素材購入費等）
  • 月額保守 = ホスティング・軽微修正・レポート等、2章で確認した保守範囲に応じて設定
  • 参考レンジ（案件ごとに調整。上振れ・下振れは工数次第で可）:
  • イベント/プロモ LP: 初期 10〜30万円、月額 5,000〜15,000円
  • コーポレートサイト: 初期 15〜50万円、月額 10,000〜20,000円
  • 個人・クリエイターサイト: 初期 5〜15万円、月額 3,000〜10,000円
  1. 見積額が固まったら、先方に金額・内容を提示し、合意を得る。
  2. 合意後、05_業務手順書/制作申込書兼契約書_ひな形.md を流用して契約書を作成する。プラン欄は「カスタム」とし、初期制作費・月額・支払方法・制作期間・検収期限・最低利用期間・解約予告期間を案件に合わせて記入する。ひな形のプラン欄はライト/スタンダード/プレミアムのチェックボックスのみのため、カスタム案件では選択肢を二重取り消し線等で無効化し、余白に「カスタム（個別見積・見積書 No.◯◯）」と記載する。版権素材がある場合は、7章（著作権・データの帰属）または特記事項に「権利処理済み素材の支給は甲（依頼主）の責任とし、乙（制作者）は当該素材の生成・複製を行わない」旨を明記する。
  3. 契約書の内容を admin /stores/[店舗ID]「契約条件」フォームにも入力し（プラン欄は custom、金額・支払方法・制作期間等は上記契約書と一致させる）、「顧客へ提示」を押す。顧客は portal /contract で内容を確認し、「内容に同意して申し込む」を押すと契約済み（contracted）に進む。この工程は飲食ラインと共通の画面・挙動を使う。

確認ポイント:

  • 契約書（書面）と admin 契約条件フォームの金額・期間が一致している。
  • 版権素材がある案件は、契約書に権利処理済み素材の支給条件が明記されている。
  • portal 側で顧客が契約に同意し、案件ステータスが「契約済み」に進んでいる。

4. Stripe Price 作成（個別見積対応）

誰が: スタッフ

画面: Stripe ダッシュボード → admin /stores/[店舗ID]「契約条件」または billing 画面

飲食ラインは共通の3プランPriceを使い回しますが、汎用ラインは案件ごとに金額が異なるため、契約が固まった段階でその契約専用の Stripe Price を作成します。

  1. Stripe ダッシュボードで商品「Musubi カスタム」（既存になければ先に商品として作成）を開く。
  2. その商品配下に、この契約の月額保守料に対応する新しい Price を作成する。種別は「継続的（Recurring）」、請求間隔は「月次」、金額は3章で確定した月額保守料（税込）を設定する。
  3. 作成した Price の ID（price_ で始まる文字列）を控える。
  4. admin の契約画面で custom プラン＋この Price ID を登録する。以後、この契約のサブスクリプション作成時にはこの Price ID が使われる。

※ admin /billing 画面の「契約を作成」フォームでプランに「カスタム（個別見積）」を選び、この Price ID と月額（円）を入力してサブスクリプションを作成する。

  1. 初期制作費（初回のみの費用）はサブスクリプションに含めず、Stripe の Invoice（単発請求）機能で別途発行する。支払方法が銀行振込の契約では、Invoice の代わりに契約書記載の振込条件に従い直接請求する。

確認ポイント:

  • Stripe ダッシュボードに、この契約専用の月額 Price が「Musubi カスタム」商品配下に作成されている。
  • Price ID を admin の契約情報に控えており、他契約の Price と取り違えていない（契約ごとに Price が1対1で対応する）。
  • 初期制作費の請求（Invoice または振込）を別途手配済み、または手配予定が決まっている。

5. 制作・公開

誰が: スタッフ

画面: admin /stores/[店舗ID] → /builder/[店舗ID]/edit

  1. ヒアリングで確認した素材・内容が揃ったら、案件詳細から制作を開始する（飲食ラインと同じ「今やること」ヒーローの導線を使う）。
  2. 汎用ラインはテンプレ＋AI生成ではなく custom モード／手組み で制作する。デザインは案件ごとの訴求・ブランドに合わせて個別に組む（イベント案件は末尾の付録チェックリストを構成の基準にする）。
  3. /builder/[店舗ID]/edit の直接編集画面、または手組みの HTML/CSS を用意して差し替える。テキスト・画像の直接編集機能は飲食ラインと共通。
  4. 画面右上の「設定」ドロワーでサブドメイン・テーマを設定する。独自ドメインを使う契約の場合は 独自ドメイン運用・移管手順.md に従い Cloudflare 側の紐付けを別途行う。
  5. 「保存して公開」でサブドメインを確定し Cloudflare Pages へデプロイする。版権素材を扱う案件は、公開前に先方の監修・確認フローがある場合、そのフローを経てから公開する。
  6. プレビューが整ったら「プレビューを顧客へ」を押す。案件ステータスが「ご確認待ち」に進み、顧客は portal /home からプレビューを確認・承認する（この工程は飲食と共通）。

確認ポイント:

  • 公開後、サブドメイン（または独自ドメイン）で実際にサイトが表示される。
  • イベント/プロモ案件は付録チェックリストの各項目が抜けなく反映されている。
  • 版権素材を扱う案件は、公開前に先方の監修・許諾が取れていることを確認済み。
  • 顧客がプレビューを承認し、admin の納品前提チェックが揃っている。

6. 課金開始・月次運用

誰が: スタッフ

画面: admin /stores/[店舗ID]「納品」セクション → portal /pay

  1. custom プランは admin の納品ボタン（振込パスを含む）ではサブスクリプションを開始できない（プラン不整合として拒否される）。契約合意・プレビュー承認・サイト公開の3条件が揃ったら、admin /billing 画面の「契約を作成」フォームからプランに「カスタム（個別見積）」を選び、4章で作成した Price ID と月額を入力してサブスクリプションを開始する。
  2. 3プラン（ライト/スタンダード/プレミアム）でカード決済の契約は、飲食と同じくトライアル付きでサブスクリプションが開始し、顧客に portal /pay でのカード登録を案内する。custom プランは billing 画面からの開始のためトライアルなしで即時課金が始まる（事前にカード登録案内を済ませておく）。銀行振込の契約は Stripe を使わず契約記録のみで運用する。
  3. 月次運用の中身は、飲食ラインの「口コミAI返信＋月次レポート」固定ではなく、3章の契約時に定義した内容（軽微な文言修正の回数、レポートの要否、更新代行の範囲等）に従う。契約書・admin のメモに記載した保守範囲を都度参照し、範囲外の依頼は追加見積とする。
  • event テンプレ案件は更新代行に加えて顧客セルフ更新も選べる。portal /site/edit の「リスト編集」タブから、開催地（venues）・ニュース（news）・グッズ（goods）・SNS リンクを顧客自身が行単位で追加・削除・編集できる（従来の「かんたん編集（インライン）」＝文字/画像のみの編集と併設）。生HTML・アイコンSVG等の運営専用項目は顧客フォームから隠蔽されており、運営側（admin）のみ編集可能なため、構造以外の込み入った変更は引き続き更新代行で対応する。event 以外のテンプレ（restaurant 系等）はこのリスト編集フォームを持たず、従来どおりインライン編集＋更新代行のみ。
  1. 未入金対応（決済失敗時のリマインド〜自動一時停止〜復帰）は飲食ラインと同じ仕組みがそのまま動く。

確認ポイント:

  • admin /billing に有効な契約（custom Price を使用）が表示される。
  • 顧客への納品完了メール・カード登録案内が届いている（カード決済の場合）。
  • 月次の保守作業が契約時定義の範囲内で実施されている。

7. 会期終了・解約処理

誰が: スタッフ（顧客はメール・電話で解約や会期終了後の扱いを申し出る）

画面: admin /stores/[店舗ID]「解約処理」セクション

  1. 期間限定サイト（イベント/プロモ案件）は、2章のヒアリングおよび3章の契約時点で決めた会期終了後の扱い（閉鎖／アーカイブとして残す／通常サイトとして月額継続）を、会期終了のタイミングで実行する。
  • 閉鎖する場合: admin の解約処理セクションから通常の解約フローを実施し、サブスクリプションを解約する。サブドメイン運用なら「サブドメイン削除」を選ぶ。
  • アーカイブとして残す場合: サイトは削除せず、サブスクリプションのみ解約する（月額保守は終了するが閲覧は可能な状態を残す旨を契約時の合意に基づき運用する）。
  • 通常サイトとして継続する場合: 解約処理は行わず、6章の月次運用にそのまま移行する。
  1. 通常の解約申し出（会期に関係なく契約終了の依頼）を受けた場合は、飲食ラインと同じ解約処理フロー（サブドメイン削除／独自ドメイン移管の選択、確認ダイアログでの実行）に従う。
  2. 独自ドメインの移管を伴う場合は 独自ドメイン運用・移管手順.md パターンAに従い、レジストラ移管・ネームサーバ確認・Cloudflareゾーン移管・移管手数料請求を手動で行う。

確認ポイント:

  • 会期終了後の扱いが契約時の合意どおりに実行されている(閉鎖/アーカイブ/継続のいずれか)。
  • 解約する場合、Stripe ダッシュボードでサブスクリプションが解約されている。
  • サブドメイン削除の場合、該当サブドメインが表示されなくなっている。
  • 顧客に解約・会期終了後の扱いについての案内が届いている。

付録: イベント告知サイト必須要素チェックリスト

実在するポップアップストア/イベント告知サイト3例の分析から抽出した、制作・検収時に確認する必須要素一覧です。5章の制作時に構成の土台とし、公開前の検収でも1点ずつ満たしているか確認します。

  1. Hero — イベント名・キービジュアル・キャッチコピーがファーストビューに揃っている。
  2. 会期 — 開始日は必須。終了日、または「順次開催」等の表記。
  3. 会場 — 店舗名・住所。複数拠点で開催する場合は一覧化されている。
  4. 営業時間 — 通常の営業時間に加え、初日・最終日等の特別日の例外が明記されている。
  5. 入場方式 — フリー入場／抽選／先着のいずれか、予約が必要な場合は予約導線が用意されている。
  6. グッズ一覧 — カテゴリ別に整理され、税込価格が明記されている。
  7. ノベルティ／購入特典 — 条件・内容・数量上限・終了条件の3点セットが揃っている（いずれか欠けると問い合わせ対応が増える）。
  8. 購入制限 — お一人様/1会計あたりの上限点数が明記されている。
  9. 注意事項 — 返品・交換不可、取り置き不可、画像はイメージである旨等の免責事項が明記されている。
  10. SNSリンク — 最新情報の告知先（公式SNSアカウント）へのリンクがある。
  11. 問い合わせ先 — 主催・運営への問い合わせ手段が明記されている。
  12. 著作権・協力表記 — フッターに主催・協力ブランド名が正しい表記で掲載されている（版権キャラクターを扱う案件は先方指定の表記ルールに厳密に従う）。

テスト実施メモ

案件を1件通しで運用する際は、各章の「確認ポイント」を上から順に満たせるかで進行を確認してください。特に4章（Stripe Price 作成)は飲食ラインにない工程のため、Price ID の控え忘れ・契約との紐付け誤りがないよう注意します。詰まった箇所・改善したい点はステップ番号とともに記録して共有してください。
