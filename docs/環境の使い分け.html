使い分けルール

  • 開発・動作確認・E2E は必ずテスト環境。 本番環境は実データのみを扱う場所であり、テストデータを作らない。
  • 切り替えは特別な操作を要しない。musubi-admin / musubi-portal ともに .env.development.local を置いてあり、Next.js は next dev 実行時に .env.local より優先してこちらを読み込む仕様。したがって ローカルで npm run dev を起動するだけで自動的にテスト環境（musubi-test）に向く。 next build や本番実行では .env.development.local を読まないため、本番の挙動には一切影響しない。
  • 本番 URL（admin.musubiweb.com / portal.musubiweb.com）に対して動作確認目的でテストアカウント作成・ダミー店舗登録などは行わない。

テスト環境の入り方

  1. cd musubi-admin && npm run dev（ポート3000）／別ターミナルで cd musubi-portal && npm run dev（ポート3001）
  2. ブラウザで http://localhost:3000 （管理アプリ）または http://localhost:3001 （ポータル）を開く
  3. テストアカウントでログイン
  • staff アカウント: staff-test@musubiweb.com
  • オーナーアカウント: owner-test@musubiweb.com（対象店舗: test）
  • パスワードは両アカウントとも musubi-admin/.env.development.local 冒頭コメント参照

migration の適用順序

新しい migration は必ずテスト → 本番の順で適用する。

  1. まずテスト環境に適用: apply_migration.mjs --ref=vqflyyoexsxqcfajwoxv <migrationファイル>
  2. テスト環境（ローカル dev）で動作確認
  3. 問題なければ本番に適用: apply_migration.mjs --ref=hsyhiiqrrtrbxexqyixi <migrationファイル>（本番適用はユーザー承認必須）

apply_migration.mjs は 06_スクリプト/apply_migration.mjs。Supabase Management API 経由で SQL を実行するスクリプトで、SUPABASE_ACCESS_TOKEN（セッション限りの手渡し）が必要。

テストデータのリセット

テスト環境のデータが壊れた・汚れた場合は、テーブルのテストデータ行（leads/stores/projects の name/template が test のもの、および staff_members・auth のテストアカウント）を削除した上で、環境変数一覧と登録手順.md のテスト環境節に記載した seed 手順（staff アカウント作成 → テスト店舗一式 leads/stores/projects → owner アカウント作成）をなぞって再投入する。スキーマ自体（migration 適用状態）は本番と揃っている前提なので、崩れるのは基本的にデータ行のみ。

Stripe テストキー未設定について

現状 STRIPE_SECRET_KEY / STRIPE_WEBHOOK_SECRET はテスト環境で空欄のため、課金まわりの dev 確認はできない（no-op）。必要になったら Stripe Dashboard をテストモードに切り替えて test キーを発行し、musubi-admin/.env.development.local に追記する。本番キー・本番 price ID とは完全に別物なので混在させない。
